# Phase 4: The Prefrontal Cortex (Flagship Layer)

This layer is strictly reserved for high-level architectural planning, complex synthesis, and final PR review. It never does manual labor or bulk triage.

## Objective
Utilize premium frontier models exclusively for logic and architecture, maximizing their output quality by feeding them only pristine, pre-filtered context.

## Tech Stack
*   **Azure OpenAI / AWS Bedrock:** Enterprise hosting for frontier models (GPT-4o, Claude 3.5 Sonnet). ($5K+ credits)
*   **Linear:** Issue tracking and project management integration. (6 months Business Plan)

## Step-by-Step Implementation

### Step 4.1: Flagship Worker Setup
1.  Create a `flagship-worker` listening to `rector.agents.flagship.request`.
2.  Connect to **Azure OpenAI** via Doppler secrets. Use standard SDKs but configure strict timeout and retry policies.

### Step 4.2: Architectural Planning Stage (`2_ARCHITECTURAL_PLAN`)
1.  When a task enters the `ARCHITECTURAL_PLAN` state, the Thalamus router invokes the Flagship model.
2.  **Input:** The "Distilled Context" generated by the Intake Agent (Chroma + Perplexity summaries).
3.  **Prompt Instruction:**
    "You are the Lead Architect. Review this feature request and codebase context. Output a strict JSON array of parallelizable sub-tasks required to implement this feature. Assign each task to an SLM role (e.g., `IMPLEMENT_LOGIC`, `WRITE_TESTS`). Do NOT write the code yourself."
4.  **Output:** A JSON array of sub-tasks. The Thalamus router saves this array to MongoDB and triggers the SLM Fan-Out phase.

### Step 4.3: Final Synthesis Stage (`6_FINAL_SYNTHESIS`)
1.  Once all SLMs have completed their tasks and passed the Deterministic Sandbox, the Thalamus router triggers the Flagship model again.
2.  **Input:** The diffs generated by the SLMs, the test results from Depot, and the original Linear issue.
3.  **Prompt Instruction:**
    "Review these verified code changes. Ensure they logically satisfy the initial feature request. Resolve any integration conflicts between the parallel diffs. Output the final, unified git patch."
4.  The output is applied to the codebase.

### Step 4.4: Linear Integration (The Human Handoff)
1.  Setup incoming webhooks from **Linear** to Make.com or directly to the Thalamus API gateway.
2.  When a developer assigns a Linear ticket to "Rector", extract the ticket description and push to `rector.tasks.ingest`.
3.  Upon `FINAL_SYNTHESIS` completion, use the Linear GraphQL API to transition the ticket to "In Review", attach the PR link, and add a comment summarizing the changes and total token cost.

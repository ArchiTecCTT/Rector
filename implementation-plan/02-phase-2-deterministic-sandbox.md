# Phase 2: The Deterministic Sandbox

The sandbox provides the programmatic guardrails. It executes AI-generated code, runs tests, and parses errors so the system can heal itself locally.

## Objective
Safely execute untrusted code generated by SLMs, validate it against unit tests and linters, and capture precise error outputs for the Healing Loop.

## Tech Stack
*   **Depot:** Blazing fast container builds and isolated execution environments. ($5K credits)
*   **CodeCov:** Test coverage monitoring. (Free public/private access)
*   **Codescene:** Code anti-pattern and hotspot analysis. (Student account)
*   **Sentry:** Error tracking and stack trace parsing. (Team features, 50k errors)

## Step-by-Step Implementation

### Step 2.1: Sandbox Worker Initialization
1.  Create a new Node.js/Python worker (`sandbox-worker`) hosted on DigitalOcean/Heroku.
2.  Connect the worker to Kafka, listening to the `rector.sandbox.validation` topic.
3.  Define standard execution environments using **Depot**. Create optimized Dockerfiles for targeted languages (e.g., Node.js with Vitest and ESLint pre-installed, cached globally via Depot's build caching).

### Step 2.2: AST Injection & Execution Logic
1.  When a validation event is received, it will contain: `task_id`, `file_path`, and `new_code_content`.
2.  The worker writes the `new_code_content` to a temporary virtual volume.
3.  Trigger a headless **Depot** build/run command that mounts the volume and executes the project's test suite (`npm run test`, `pytest`, etc.).
4.  Capture `stdout`, `stderr`, and the exit code.

### Step 2.3: Quality Gates Integration
1.  **CodeCov:** If the tests pass (exit code 0), run the coverage reporter. If coverage drops below the required threshold, treat it as a failure.
2.  **Codescene:** Trigger a CLI check against the file. If Codescene flags a new critical anti-pattern or complex hotspot, treat it as a failure.

### Step 2.4: The Healing Loop Routing (Sentry)
1.  **On Success:** Emit a `SUCCESS` payload to `rector.sandbox.result`. The Thalamus router updates the state machine and moves to final synthesis.
2.  **On Failure:**
    *   Pipe the `stderr` stack trace directly into **Sentry**.
    *   Use Sentry's parsing logic (or a lightweight regex script) to extract the exact file name, line number, and error type (e.g., `TypeError`, `SyntaxError`).
    *   Extract the specific Abstract Syntax Tree (AST) node or line from the code.
    *   Emit a `FAILED` event to `rector.sandbox.result` containing: `error_message`, `line_number`, and `code_snippet`.
    *   The Thalamus router will catch this, transition to `5_HEALING_LOOP`, and route the tiny payload back to an SLM Fixer agent for pennies.

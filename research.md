# Research: Production LLM model discovery for BYOK provider setup

## Summary
Production BYOK setup should not use one generic model-list path. Rector should implement provider-specific discovery adapters: OpenAI-compatible `/v1/models` where supported, Azure OpenAI resource/deployment discovery with Azure-specific data/control-plane split, AWS Bedrock regional control-plane discovery, and Cloudflare Workers AI account catalog search. Treat discovery as advisory: cache it, expose manual refresh, and validate the selected model/deployment with a cheap per-capability probe before saving.

## Findings
1. **OpenAI-compatible providers usually expose `GET /v1/models`, but metadata quality varies.** OpenAI’s endpoint lists currently available models with `id`, `created`, `object`, and `owned_by`; Together’s OpenAI compatibility layer maps `models.list` / `models.retrieve` to `GET /v1/models`, while its native `GET /models` returns richer metadata such as `type`, `display_name`, `context_length`, and pricing. Rector should support both: first try provider-native catalog when known, then OpenAI-compatible `/v1/models` for custom base URLs. [OpenAI](https://platform.openai.com/docs/api-reference/models/list), [Together model list](https://docs.together.ai/reference/models), [Together OpenAI compatibility](https://docs.together.ai/docs/inference/openai-compatibility)

2. **Together AI is a good example of “OpenAI-compatible, not OpenAI-identical.”** Together supports chat, completions, embeddings, image, speech, transcription, and model list through OpenAI-compatible routes, but not the Responses API, Assistants, Threads, OpenAI-shaped Batch, or OpenAI-shaped fine-tuning. Its model IDs are namespaced, and OpenAI model strings like `gpt-4o` return 404. Rector should record endpoint capability flags per provider/model and avoid assuming `/v1/responses` or exact OpenAI parameter behavior just because `/v1/models` works. [Together OpenAI compatibility](https://docs.together.ai/docs/inference/openai-compatibility)

3. **Azure OpenAI discovery has two separate concepts: available base models and deployed resources.** The Azure OpenAI data-plane API `GET {endpoint}/openai/models?api-version=2024-10-21` returns models accessible to the Azure OpenAI resource and includes capabilities such as `chat_completion`, `completion`, `embeddings`, `fine_tune`, `inference`, lifecycle status, and deprecation data. This does not replace deployment discovery: user traffic is normally sent to an Azure deployment name, not directly to a base model ID. [Azure models list](https://learn.microsoft.com/en-us/rest/api/azureopenai/models/list?view=rest-azureopenai-2024-10-21)

4. **Azure deployment listing needs the Azure management plane, not just an endpoint API key.** The management API lists deployments for a Cognitive Services account at `GET https://management.azure.com/subscriptions/{subscriptionId}/resourceGroups/{resourceGroupName}/providers/Microsoft.CognitiveServices/accounts/{accountName}/deployments?api-version=2025-06-01`, returning deployment `name`, SKU/capacity, provisioning state, and underlying model `{format,name,version}`. For BYOK UX, if the user only supplies Azure endpoint + API key, Rector may be able to list accessible base models but should ask for deployment name(s) or probe a named deployment; full deployment auto-discovery requires subscription ID, resource group, account name, and Azure auth. [Azure deployments list](https://learn.microsoft.com/en-us/rest/api/aifoundry/accountmanagement/deployments/list?view=rest-aifoundry-accountmanagement-2025-06-01)

5. **Azure regional/capacity constraints are first-class.** Azure exposes a location-based model capacity API: `GET /subscriptions/{subscriptionId}/providers/Microsoft.CognitiveServices/locations/{location}/modelCapacities?...&modelFormat={modelFormat}&modelName={modelName}&modelVersion={modelVersion}`, returning SKU capacity and available fine-tune capacity for a model/version/location. Rector should persist Azure `location`, deployment name, model name/version, SKU, and provisioning state; show “region/account may not support this model/version” separately from “invalid key.” [Azure model capacities](https://learn.microsoft.com/en-us/rest/api/aifoundry/accountmanagement/location-based-model-capacities/list?view=rest-aifoundry-accountmanagement-2025-06-01), [Azure region support](https://learn.microsoft.com/en-us/azure/foundry/reference/region-support)

6. **AWS Bedrock discovery is regional by construction.** `ListFoundationModels` lists foundation models usable in the AWS region targeted by the Bedrock control-plane client and supports filters by provider, output modality, inference type, and customization type. Results include `modelId`, `providerName`, input/output modalities, streaming support, customizations, inference types, and lifecycle. Rector should require/select AWS region before listing models and key the cache by account + region. [Bedrock ListFoundationModels](https://docs.aws.amazon.com/bedrock/latest/APIReference/API_ListFoundationModels.html)

7. **Bedrock model visibility does not fully prove account access or invoke readiness.** AWS provides `GetFoundationModelAvailability(modelId)`, which returns `authorizationStatus`, `entitlementAvailability`, and `regionAvailability` values such as `AUTHORIZED` / `NOT_AUTHORIZED` and `AVAILABLE` / `NOT_AVAILABLE`. Rector should call this for the selected model and still run a minimal invoke probe, because IAM, model access agreements, quotas, and inference profile requirements can fail after catalog listing. [Bedrock GetFoundationModelAvailability](https://docs.aws.amazon.com/bedrock/latest/APIReference/API_GetFoundationModelAvailability.html)

8. **Bedrock cross-region inference needs explicit UX.** AWS documents cross-region inference profiles with source regions and destination regions; some profiles can route requests to multiple destination regions, and SCP/IAM restrictions can make the request fail if any destination region is blocked. Rector should distinguish “in-region model ID” from “inference profile ID,” display possible routing/data-residency implications, and store source region + profile ID + destination-region metadata when available. [Bedrock inference profiles](https://docs.aws.amazon.com/bedrock/latest/userguide/inference-profiles-support.html), [Bedrock supported models](https://docs.aws.amazon.com/bedrock/latest/userguide/models-supported.html)

9. **Cloudflare Workers AI has an official account-scoped model catalog and currently reports 78 models.** The public docs page says “We found 78 models” and includes task/type tags such as Text Generation, Text Embeddings, Text-to-Image, ASR, TTS, Translation, Text Classification, deprecated, batch, real-time, LoRA, function calling, reasoning, and vision. Programmatic discovery is via Cloudflare API `GET /accounts/{account_id}/ai/models/search`, with SDK method `client.ai.models.list(...)`; a schema endpoint `GET /accounts/{account_id}/ai/models/schema` returns model input/output schema. Rector should filter catalog results to chat/text-generation and embeddings by default, hide deprecated models unless “show deprecated” is enabled, and use schema metadata for capability probes. [Workers AI models](https://developers.cloudflare.com/workers-ai/models/), [Cloudflare AI models API](https://developers.cloudflare.com/api/node/resources/ai/subresources/models/)

10. **Recommended Rector design: normalized discovery with provider-specific adapters.** Use a common internal shape like `{provider, accountScope, region, endpoint, modelId, deploymentId, displayName, modality, capabilities, contextWindow, pricing, lifecycle, deprecation, requiresDeployment, requiresRegion, source}`. Adapter behavior: OpenAI-compatible = `GET /v1/models`; Together = native `GET /models` preferred plus `/v1/models` fallback; Azure = data-plane models + optional ARM deployments/capacity when management credentials exist; Bedrock = regional `ListFoundationModels` + selected-model availability; Cloudflare = account catalog search + schema lookup.

11. **Caching should be short enough for fast-moving catalogs but long enough to avoid slow setup.** Practical TTLs: static/provider public catalog fallback 24h; credential/account/region-scoped dynamic discovery 15-60 minutes; Azure deployments and Bedrock availability 5-15 minutes; negative/error results 30-120 seconds; per-model successful probe 5-15 minutes. Always show `last refreshed`, provide manual refresh, and invalidate cache on provider, key, endpoint, region, account ID, subscription/resource group/account, or base URL change.

12. **Test-connection UX should be staged, precise, and cheap.** Recommended flow: (a) validate required fields and endpoint URL shape locally; (b) authenticate/list catalog if available; (c) let user pick model/deployment with capability labels; (d) run a minimal probe for the chosen capability: one-token chat completion for chat models, tiny embedding for embedding models, or provider-native dry/cheap request where available; (e) classify errors as auth, endpoint/base URL, region/location unsupported, deployment not found, model access/agreement missing, quota/rate limit, parameter incompatibility, or content/safety. Save only after the selected model/deployment passes or the user explicitly chooses “save unverified.”

## Gaps
- Provider docs do not define a universal cache TTL; TTL guidance above is practical design advice, not a standard.
- Azure BYOK discovery depth depends on credential type. Endpoint + API key is enough for data-plane model listing/probing, but deployment auto-discovery and capacity checks require Azure management-plane identity and resource metadata.
- Actual invoke probe payloads should be implemented per provider/model capability and tested against Rector’s provider adapters; catalog success alone is insufficient.
